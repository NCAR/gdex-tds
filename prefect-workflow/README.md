# prefect-workflow

Prefect-based automation workflows for managing datasets on the THREDDS Data Server (TDS).

---

## Workflows

### 1. `auto_add_data_tds.py` — Daily autoscan (runs at 1:00 AM)

Scans the RDA metadata database for new datasets and adds them to the TDS catalog if their format is supported.

**Steps:**
1. Query all dataset IDs from the RDA metadata database (`rdadb`)
2. Query all dataset IDs currently listed in `rda-tds/content/catalog.xml`
3. For each dataset in the database not yet in the catalog:
   - Check if its format is supported (NetCDF, GRIB1, GRIB2)
   - If supported: generate `catalog_d<dsid>.xml` via `src/createXML.py` and insert it into `catalog.xml` in alphabetical order
4. Write a timestamped log file `auto_add_data_tds_<YYYY-MM-DD-HH_MM_SS>.log` listing all newly added dataset IDs

**Output log format** (one line per added dataset):
```
[2025-11-14-12_00_38] - d014619 added
....
```

**Cron runner:** `~/cron/tds/gdex-tds-data-autoscan.sh`
- PBS job on NCAR Casper (`#PBS -q gdex`, 1 CPU, 10GB, 30 min walltime)
- Stashes any local git changes before running, restores on exit (trap cleanup)
- On success: commits updated `catalog.xml`, `catalog_*.xml`, and log file to a new branch `auto_add_data_tds_<YYYY-MM-DD>`, pushes to GitHub
- GitHub Actions workflow `auto-merge-data-tds.yml` auto-creates a PR against `main` and auto-merges it, notifying `@chiaweh2`
- On failure: sends email to `$USER@ucar.edu`

---

### 2. `add_control_tds.py` — Post-merge control check (runs at 6:00 AM)

Runs after `auto_add_data_tds.py` has completed, been committed, and merged to `main`. Verifies the new datasets are properly registered and reachable on the live TDS.

**Steps:**
1. Receives the log file from today's autoscan run as argument (`auto_add_data_tds_<date>*.log`)
2. Parses the log file to extract the list of newly added dataset IDs
3. For each dataset ID:
   - Generates a new CTL index via `src/createCTL.py <dsid>`
   - Retrieves the existing control file template via `dsrqst -gc -ds <dsid>`
   - Registers the TDS URL entry in the database via `dsrqst -sc -ds <dsid> -if <control_file> -md -nc`
   - Validates the TDS dataset URL is reachable (HTTP status 200–399)

**Why it runs at 6:00 AM (5 hours after autoscan):**
- Autoscan runs at 1:00 AM → commits and pushes new branch
- GitHub Actions auto-creates PR and merges to `main` (~minutes)
- Argo CD detects the `main` update and redeploys TDS pod (~minutes)
- TDS pod restarts and loads the new catalog (~minutes)
- 6:00 AM gives sufficient buffer for all steps to complete before the URL check

**Cron runner:** `~/cron/tds/gdex-tds-add-control.sh`
- PBS job on NCAR Casper (`#PBS -q gdex`, 1 CPU, 1GB, 30 min walltime)
- Looks for today's log file `auto_add_data_tds_<YYYY-MM-DD>*.log` in `prefect-workflow/`; exits with error if not found
- On failure: sends email to `$USER@ucar.edu`

## Log Files

| File | Generated by | Purpose |
|------|-------------|---------|
| `auto_add_data_tds_<datetime>.log` | `auto_add_data_tds.py` | Lists dataset IDs added to TDS catalog |
| `add_control_tds_<datetime>.log` | `add_control_tds.py` | Lists dataset IDs with TDS URL registered |

---

## Full Daily Pipeline

```
01:00 AM  gdex-tds-data-autoscan.sh (PBS)
            └─ auto_add_data_tds.py
                 └─ scan DB → update catalog XML → write log
                      └─ git push branch → GitHub Actions → auto PR → auto merge → main
                           └─ Argo CD detects main change → redeploy TDS pod

06:00 AM  gdex-tds-add-control.sh (PBS)
            └─ add_control_tds.py
                 └─ parse today's log → create CTL → register TDS URL → verify URL reachable
```
